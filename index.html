This repository provisions, configures, and deploys a secure web + monitoring stack on AWS using Terraform and Ansible. It demonstrates a complete DevOps lifecycle from Infrastructure as Code (IaC) to automated deployment and observability.

1. Architecture Overview

All infrastructure is provisioned in the ap-southeast-1 (Singapore) region, designed with security and scalability in mind.

Key Components

  • State Management: Terraform state is stored securely in an S3 Backend (devops-bootcamp-terraform-mohdadlijaaffar) with encryption enabled.
  • Network: A custom VPC with segregated Public (Web) and Private (Ansible/Monitor) subnets.
  • Security:
    • Security Groups restricted to necessary ports (80, 9100, 22).
    • No Public IP for Monitoring or Ansible servers.
    • Access managed via AWS Systems Manager (SSM).
  • Application: Dockerized web app deployed via ECR.

Fig 1: Managing nodes securely via AWS Systems Manager.

2. Quick Start

Ensure you have terraform, ansible, and aws-cli installed.

Environment Setup

aws configure --profile default
export AWS_REGION=ap-southeast-1

3. Infrastructure & Configuration

3.1 Terraform (IaC) 30% Coverage

The infrastructure is fully automated. Key configuration highlights:

  • VPC & Routing: Public subnet uses an Internet Gateway; Private subnet uses a NAT Gateway for secure outbound updates.
  • Compute: for_each loops used to standardize t3.micro instances with IAM roles attached.

3.2 Ansible (Config Management) 25% Coverage

Automation is handled by the Ansible Controller in the private subnet:

  • Install Docker: Automated installation of Docker Engine on all nodes.
  • Web Deployment: Pulls image from ECR, handles ECR Login via AWS CLI, and deploys the app.
  • Monitoring Setup: Deploys Node Exporter, Prometheus, and Grafana containers.

4. Monitoring & Observability 20% Coverage

The stack provides real-time insights into infrastructure health.

Metrics Collected

Prometheus scrapes the Web Server (via Node Exporter on port 9100) for:

  • CPU Usage
  • Memory (RAM) Usage
  • Disk Usage

Visualization

Grafana visualizes these metrics using the "Node Exporter Full" dashboard. Access is secured via a Cloudflare Tunnel, ensuring the monitoring server remains private with no open inbound ports.

5. Bonus Achievements +10% Bonus

Beyond the core requirements, the following advanced features were implemented:

1. CI/CD Pipeline: GitHub Actions configured to build Docker images and push to ECR automatically on commit.
2. Self-Hosted Runner: Setup a local GitHub Runner for direct deployment pipelines (active on-demand).
3. Zero-Trust Access: Implemented AWS Systems Manager (SSM) for shell access, eliminating the need for SSH keys or Bastion hosts.
4. Cloudflare Tunnel: Monitoring server is completely isolated from the public internet, accessible only via secure tunnel.

6. Dev Diary: Challenges & Solutions

Real-world battles fought during this deployment.

Challenge 1: The "AWS Not Found" Error

Problem: Ansible failed at ECR Login because the vanilla Ubuntu AMI lacks the AWS CLI.
Fix: Updated playbook to install dependencies first.

- name: Download AWS CLI
  get_url:
    url: "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" ...

- name: Install AWS CLI
  command: /tmp/aws/install --update

Challenge 2: The "Stale Image" Persistency

Problem: The Web Server refused to show the new version of the app and stuck to the old Docker image.
Fix: The "Fresh Start" Protocol. I updated the playbook to explicitly stop and remove the old container before starting the new one.

Challenge 3: The "Localhost" Monitoring Trap

Problem: Grafana running in Docker couldn't reach Prometheus via localhost:9090.
Fix: Configured Grafana to use the Private IP of the host server.

http://10.0.0.136:9090
