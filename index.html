Setup, Deployment & Monitoring Guide. v1.4.0

This repository provisions, configures, and deploys a small web + monitoring stack on AWS using Terraform and Ansible. It serves as a proof-of-concept for secure, private networking with public exposure via load balancers and tunnels.

1. Architecture Overview

All infrastructure is provisioned in the ap-southeast-1 (Singapore) region.

  • Infrastructure: Terraform (VPC, Subnets, EC2, IAM, ECR).
  • Configuration: Ansible (Docker installation, App deployment, Monitoring setup).
  • Application: Dockerized web app with custom environment variables.
  • Monitoring: Prometheus & Grafana (node_exporter).

2. Quick Start

Ensure you have terraform, ansible, and aws-cli installed.

Environment Setup

aws configure --profile default
export AWS_REGION=ap-southeast-1

3. Dev Diary: Challenges & Solutions

Real-world battles fought during this deployment.

Challenge 1: The "AWS Not Found" Error

Problem: Ansible failed at ECR Login because the vanilla Ubuntu AMI lacks the AWS CLI.
Fix: Updated playbook to install dependencies first.

- name: Download AWS CLI
  get_url:
    url: "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" ...

- name: Install AWS CLI
  command: /tmp/aws/install --update

Challenge 2: The "Stale Image" Persistency

Problem: The Web Server refused to show the new version of the app and stuck to the old Docker image.
Fix: The "Fresh Start" Protocol. I updated the playbook to explicitly stop and remove the old container before starting the new one.

- name: Stop running container
  command: "docker stop my-app"
  ignore_errors: yes

- name: Remove old container
  command: "docker rm my-app"
  ignore_errors: yes

Challenge 3: The "Localhost" Monitoring Trap

Problem: Grafana running in Docker couldn't reach Prometheus via localhost:9090.
Fix: Configured Grafana to use the Private IP of the host server.

http://10.0.0.136:9090

4. Infrastructure & Configuration

4.1 Infrastructure (Terraform) IaC

The infrastructure code in terraform/ fully automates the AWS environment. Key design decisions include:

A. Network Segmentation (vpc.tf)

I architected a secure VPC (10.0.0.0/24) with strict segregation:

  • Public Subnet (10.0.0.0/25): Hosts the Web Server and NAT Gateway. It has a direct route to the Internet Gateway (IGW).
  • Private Subnet (10.0.0.128/25): Hosts the Ansible Controller and Monitoring Server. These instances have NO public IPs. They access the internet (for updates/Docker pull) solely via the NAT Gateway.

B. Compute Strategy (ec2.tf)

I used a for_each loop to modularize the instance creation, ensuring standard tagging and config:

  • Static Private IPs: Hardcoded IPs (10.0.0.5, 10.0.0.135, 10.0.0.136) were used to make the Ansible Inventory predictable.
  • IAM Integration: Every instance is attached to a devops-ssm-role (defined in iam.tf). This grants AmazonSSMManagedInstanceCore permissions, allowing me to console into private servers without needing a Bastion Host or SSH keys.

C. Identity & Access (IAM & Keys)

To secure access to the infrastructure and container registry, I provisioned specific identity resources:

  • ECR Management User (ecruser): A dedicated IAM user created to handle ECR operations (push/pull permissions) separately from the root account.
  • SSH Key Pair (adli-my-key): A securely generated key pair used for SSH access to the instances when SSM is not sufficient (e.g., initial debugging).

4.2 Configuration (Ansible) Automation

Once Terraform provisions the servers, Ansible takes over to configure them. The setup is broken down into modular playbooks:

A. Controller Setup

The ansible.cfg is configured to disable host_key_checking. This is critical for automation, preventing the script from hanging on the "Are you sure you want to connect?" prompt.

B. Web Server Deployment (deploy_web.yml)

This playbook handles the full application lifecycle:

  • Dependency Management: Manually downloads and installs the AWS CLI v2 using unzip and the official Amazon binary.
  • ECR Authentication: Uses the installed CLI to authenticate Docker with Amazon ECR.
  • Deployment: Pulls the latest image and restarts the container using --restart always for high availability.
  • Monitoring Sidecar: Deploys node-exporter alongside the app to expose system metrics (CPU, RAM, Disk) to Prometheus.

C. Monitoring Stack (deploy_monitor.yml)

This playbook sets up the observability layer on the private server:

  • Configuration: Copies a custom prometheus.yml that targets the Web Server's private IP (10.0.0.5:9100).
  • Containers: Launches Prometheus (Data collection) and Grafana (Visualization) via Docker.

5. How to Run

Step 1: Provision Infrastructure

cd terraform
terraform init
terraform plan -out=plan.tfplan
terraform apply "plan.tfplan"

Step 2: Configure Servers (Ansible)

Run these from the Ansible Controller node.

# 1. Install Docker
ansible-playbook -i inventory.ini install_docker.yml

# 2. Deploy Web App
ansible-playbook -i inventory.ini deploy_web.yml

# 3. Setup Monitoring
ansible-playbook -i inventory.ini deploy_monitor.yml
